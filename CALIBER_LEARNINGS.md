# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[convention]** SugarCraft PHP namespaces nest the component name twice: the autoloadable class for a library `sugarcraft/<lib>` lives at `SugarCraft\<Lib>\<Component>\<Component>` (e.g. `SugarCraft\Bits\Spinner\Spinner`, `SugarCraft\Charts\Sparkline\Sparkline`, `SugarCraft\Stickers\Flex\FlexBox`). Top-level shortcuts like `SugarCraft\Bits\Spinner` won't resolve — always use the fully-nested form when writing `use` statements or doc references.
- **[pattern]** When manually editing `composer.json` to add multiple sugarcraft packages, run `composer update <pkg1> <pkg2> ... --with-all-dependencies` rather than `composer require pkg1 pkg2 ...`. The `update` form respects all the manually-edited entries (incl. transitives added to fix "could not be found" errors) without re-running the resolver from scratch on each package.
- **[fix]** `composer require psalm/psalm` fails with "could not be found in any version" — the package name on Packagist is `vimeo/psalm`, not `psalm/psalm`. Use `composer require --dev vimeo/psalm` instead. (Composer's error message helpfully suggests the correct vendor.)
- **[gotcha]** PHP-Parser 5 (used in this repo) removed `PhpParser\Node\Stmt\Throw_` — `throw` is now an expression only. Use `Node\Stmt\Expression` whose `->expr` is `Node\Expr\Throw_`. Any `instanceof Node\Stmt\Throw_` check is dead code and triggers `phpstan class.notFound`.
- **[gotcha]** In PHP-Parser 5, `Node\VarLikeIdentifier` extends `Node\Identifier`. So `instanceof Node\Identifier` already matches both — a separate `instanceof Node\VarLikeIdentifier` branch placed after `Identifier` is unreachable (phpstan flags it `instanceof.alwaysFalse`). If you need the VarLikeIdentifier-specific behaviour, check the subclass FIRST, then `Identifier`.
- **[convention]** Static analysis runs at phpstan level 6 (`phpstan.neon`) and psalm `errorLevel=6` (`psalm.xml`), both scoped to `src/` only with `vendor/` and `tests/Fixtures/` excluded. Don't bump levels or widen scope without coordination — fixtures intentionally include malformed PHP to exercise parser error paths.
- **[pattern]** `vendor/bin/phpstan analyse --error-format=json` followed by piping into a small Python `json.loads` script gives a much more compact view of errors than the default formatter (which wraps lines at 80 cols and adds blog-link footers). Useful when triaging dozens of errors at once.
- **[fix]** Psalm's `ParadoxicalCondition` (psalm/089) and downstream `InvalidCast` (`never cannot be cast to int`) on repeated `array_key_exists($k, $data)` checks across separate `if` blocks are caused by Psalm narrowing the array shape after the first positive check (it then considers later positive checks contradictory). Fix: cache existence once into local booleans (`$hasMin = array_key_exists('min_block_size', $data); $hasMax = array_key_exists('max_block_size', $data);`) and reuse those in subsequent guards. Psalm treats the booleans opaquely and stops fighting the narrowing.
- **[pattern]** Smoke-test `--validate-config` and similar config-gating CLI flags via a tiny inline shell script that writes a JSON to `mktemp --suffix=.json`, runs the command, prints `exit: $?`, then mutates the file for invalid/missing-flag cases. Inspecting exit codes per branch (0 valid / 2 invalid / 2 misuse) catches regressions in argv plumbing that PHPUnit can't see.
- **[pattern]** When wiring a new layered config source into `Phpdup\Cli\ConfigLoader::load()` (e.g. profile excludes), pass it as a separate named parameter and apply it ONLY when the user-facing source (`--config` JSON or matching `$overrides[$key]`) is empty. Excludes specifically resolve as `!empty($data['exclude']) ? $data['exclude'] : ($profileExclude ?? $base->exclude)` — keep profiles as a *default upgrade*, not a forced override, so `--config` and explicit CLI flags continue to win.
- **[convention]** Profile JSON files in `/home/sites/php-dup-finder/profiles/*.json` use a `_description` key for human-readable docs that `ProfileRegistry::load()` strips before returning, because the validator rejects unknown keys. When adding a new profile, keep `_description` for humans but never reference it from code — it won't survive the loader.
- **[convention]** Profile JSON files use `kinds` as the user-facing key but `Config` / `$overrides` use `allowed_kinds` internally. The mapping happens in `Command::profileToOverrides()` — when adding a new profile-mappable key, mirror that translation rather than passing the JSON dict straight through to `ConfigLoader`.
- **[gotcha]** When phpunit reports `OK (266 tests)` but `phpstan analyse` then prints `[ERROR] Found 2 errors`, the chained `&& ... | tail -5` swallows the actual error lines — phpstan's table formatter pushes the bodies above the `tail -5` window. Re-run `vendor/bin/phpstan analyse 2>&1 | tail -50` (or use `--error-format=json`) to see what failed; never declare "all clean" off a tail-5 of a chained run.
- **[pattern]** When adding a new disk-cache layer to phpdup (e.g. `IndexStore`, `ClusterCache`), gate enablement behind `$config->incremental` AND exclude modes where the cache layer adds nothing — `ClusterStage` excludes `exactOnly` because hash-bucket clustering is already O(n). Build the cache config-key from the fields that actually drive output (similarity/tree thresholds, optional-blocks tunables, `exactOnly`); mismatched key invalidates wholesale. Strip heavy `PhpParser` AST nodes from cluster payloads before `serialize()` (set `$copy->generalizedAst = null` on a clone) — they bloat the cache by ~10x and are reconstructed on demand by `RefactorStage`.
- **[pattern]** Smoke-test new disk-cache layers with this 4-step shell: `rm -rf .phpdup-cache; php bin/phpdup analyze <fixture> --json /tmp/c1.json; php bin/phpdup analyze <fixture> --json /tmp/c2.json; diff /tmp/c1.json /tmp/c2.json && ls .phpdup-cache/`. The first run populates the cache; the second must hit it (look for `reused N clusters from cache` or equivalent log line); the diff must be empty (deterministic output across cache hit/miss); the `ls` confirms the new cache file lives alongside existing ones (`*.idx`, `*.cache`, `clusters.list`).
- **[gotcha]** `serialize()` of `Cluster` objects round-trips fine with `unserialize($blob, ['allowed_classes' => true])` only because `Cluster` is `final` with public mutable properties. If you mark fields `readonly`, `unserialize` cannot rehydrate them through the normal property-write path and you'll need to switch to `__serialize`/`__unserialize` explicitly.
- **[gotcha]** When extending `src/Normalization/Normalizer.php`'s `CanonicalizingVisitor::enterNode`, mode-gating is done by **early return** (`if ($mode === 'strict') return null;` then literal canonicalization, then `if ($mode !== 'aggressive') return null;` then name canonicalization). Adding a new canonicalizer means picking the right slot: `default`-tier helpers go between the strict-return and the aggressive-return; `aggressive`-only helpers go after the aggressive-return guard. Tests must cover both directions — that the helper fires in its mode, and that strict/lower modes leave the input untouched (see `testStrictModeDoesNotReorderNamedArgs` and `testDefaultModeKeepsAttributesDistinct` patterns in `tests/Unit/Normalization/NormalizerTest.php`).
- **[gotcha]** PHP-Parser 5 doesn't let you change a node's runtime class in place — `Node\Expr\Match_` cannot be morphed into `Node\Stmt\Switch_` by mutation. To make `match` and `switch` cluster together, mutate the existing node's *children* into a uniform shape (e.g. OR-fold multi-cond `MatchArm`s) and accept that the wrapper-node-name will still differ in the structural hash. Don't try to swap the class via reflection; the parent traverser will reject it.
- **[pattern]** `PatternRecognizer` test fixtures must use real parsed `Block`s, not null-AST stubs — `isSqlBuilder`/`isValidationChain`/`isQueryBuilderChain` traverse `$m->ast` via `NodeFinder` and crash on null. Build the cheapest valid fixture with `(new BlockExtractor(minSize: 1))->extract('virtual.php', (new AstParser())->parseCode('<?php function name() { return 1; }'))` (see `tests/Unit/Refactor/PatternRecognizerTest::dummyMembers()`), then mutate `$block->class`/`$block->file`/`$block->namespace` directly to drive the framework-aware tag detectors (`controller-action`, `migration`, `eloquent-model`, `repository-method`, `event-listener`, `service-provider`).
- **[convention]** `Block` properties (`class`, `file`, `namespace`, `name`, `ast`) are publicly mutable — tests deliberately overwrite them post-extraction to exercise pattern-recognizer branches keyed off file paths or class-name suffixes. Keep them mutable; if you ever propose marking them `readonly`, every reporter/pattern test in `tests/Unit/Refactor/` will need to be rewritten.
- **[gotcha]** PHPStan's `varTag.differentVariable` rule fires when a `/** @var X $foo */` PHPDoc tag is placed immediately above a `foreach ($foo->stmts as $s)` — PHPStan compares the tag's variable name (`$foo`) to the *foreach loop variable* (`$s`), not the iteree, so it reports `Variable $foo in PHPDoc tag @var does not match any variable in the foreach loop: $s`. Renaming `$foo` doesn't help (any name that isn't the inner loop variable triggers the rule). Fix: drop the `@var` tag and narrow with `if (!$foo instanceof X) continue;` at the top of the outer loop body — this gives PHPStan the same type info without violating the rule. Pattern: prefer `instanceof`-narrowing over inline `@var` tags in code that iterates `NodeFinder::findInstanceOf()` results, because phpstan.neon disables the inline-`@var` escape hatch ("Do not use assert() or inline @var PHPDoc tag to override PHPStan's inferred type").
- **[gotcha]** When adding a new pattern tag in `src/Refactor/PatternRecognizer.php`, the Golden suite (`tests/Golden/*.json`) will drift on any fixture whose cluster matches the new tag — e.g. adding `sql-query` made `tests/Golden/sql-exact.json` fail because `pattern_tags` grew an entry. Regenerate with `UPDATE_SNAPSHOTS=1 vendor/bin/phpunit --testsuite Golden`, then re-run the full suite. The same regen pass also normalizes cosmetic int→float diffs (`'similarity' => 1` vs `'similarity' => 1.0`) that surface when var_export sees a float-typed property holding an integer value.
- **[gotcha]** Adding a new readonly property to `Phpdup\Cli\Config` requires updating BOTH the constructor parameter list AND the `withOverrides()` (or any other clone-with-overrides) method's named-argument list — otherwise the field silently resets to its default whenever per-directory overrides merge in, masking the bug until a `.phpdup.json` is encountered. Search for every named-argument `Config(` constructor call when adding a property.
- **[gotcha]** `src/Parallel/PreprocessWorker.php` caches per-config tooling (extractor + normalizer + ngram fingerprinter) keyed by a `sprintf()` fingerprint of the Config fields that affect normalization output. When adding a new `Config` field that the Normalizer reads (e.g. `normalizationPlugins`), append it to the cache-key `sprintf()` — otherwise stale tooling is reused across configs differing only by the new field, producing wrong-canonical-form blocks under per-directory overrides.
- **[pattern]** Adding a new top-level config JSON key (e.g. `normalization`) requires THREE updates to `src/Cli/ConfigLoader::validate()`: append the key to the allowed-keys whitelist, add a per-key validation block (object type-check + nested-keys whitelist), and call the existing `$assertListOfStrings()` helper on any list-of-string sub-fields. Mirror the existing `report` block — same shape, same nested-key strictness.
- **[pattern]** Test SIGINT (and similar pcntl) handlers by sending the signal to the current process: `posix_kill(posix_getpid(), SIGINT); usleep(10_000);` then assert the side-effect on the shared state object. Skip the test cleanly with `markTestSkipped` when `pcntl_signal` or `posix_kill` are unavailable. Async signals (`pcntl_async_signals(true)`) dispatch immediately so no manual `pcntl_signal_dispatch()` call is needed in the test — the `usleep` is just a paranoia margin.
- **[pattern]** Tests for analyzers that consume `Block::$ngramBag` (e.g. `CoherenceAnalyzer`, anything calling `JaccardSimilarity::similarity()`) can skip the full extractor pipeline: `$stmts = (new AstParser())->parseCode('<?php $x;'); $b = new Block(file: 'x.php', range: new LineRange(1, 1), kind: 'method', namespace: null, class: null, name: null, ast: $stmts[0]); $b->ngramBag = $bag;` is enough because `Block`'s ngram-related properties are publicly mutable. Faster than re-running `BlockExtractor` + `NgramFingerprint` per fixture.
- **[convention]** `Block` properties (`class`, `file`, `namespace`, `name`, `ast`, `ngramBag`) are publicly mutable — tests deliberately overwrite them post-extraction to exercise pattern-recognizer / coherence branches keyed off file paths, class-name suffixes, or n-gram shape. Keep them mutable; if you ever propose marking them `readonly`, every reporter/pattern test in `tests/Unit/Refactor/` and `tests/Unit/Reporting/` will need to be rewritten.
- **[gotcha]** ANY change to `JsonReporter::build()`'s cluster-element shape (new key like `outlier_members`, new top-level `summary` field, etc.) breaks the Golden suite because `tests/Golden/*.json` snapshots the full structure. Same goes for adding a `Cluster` property whose default `var_export`s differently from the prior shape (e.g. a float-typed property holding an int → `1` vs `1.0`). Regenerate with `UPDATE_SNAPSHOTS=1 vendor/bin/phpunit --testsuite Golden`, eyeball the diff to make sure only the intended keys changed, then re-run the full suite + `phpstan analyse`. The same regen pass also normalizes cosmetic int→float diffs that surface when the Golden snapshot was authored before a float-typed property existed.
- **[pattern]** When piping cluster results through a post-Ranker enrichment stage that mutates `Cluster` properties (e.g. `CoherenceAnalyzer::analyze()` populating `$outlierMemberIds`), run it inside `ReportStage::execute()` AFTER `Ranker::rank()` — Ranker filters and reorders, so doing the enrichment before wastes work on clusters about to be dropped, and breaks downstream reporters that rely on indices in the final ordering.
