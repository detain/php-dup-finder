# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** Standalone `:null` return type (e.g. `function foo(): null`) is a PHP 8.2+ feature. On PHP 8.1 it's a parse error. For overrides like `NodeVisitorAbstract::enterNode` that return null, drop the explicit return type entirely (`function enterNode(Node $n) { ... return null; }`) — this works on 8.1+ and the parent declares no return type.
- **[env]** Sugarcraft libraries (`sugarcraft/candy-core`, `candy-kit`, `candy-sprinkles`, `candy-palette`, etc.) are NOT published on Packagist — `repo.packagist.org/p2/sugarcraft/<lib>.json` returns 404 for most of them. To depend on them in composer.json: add a `"repositories"` block of `{"type": "vcs", "url": "https://github.com/sugarcraft/<lib>"}` entries, set `"minimum-stability": "dev"`, and use `"dev-master"` constraints. The monorepo at `/home/sites/sugarcraft` exists for local browsing but published consumers should pull from the per-lib `github.com/sugarcraft/*` repos.
- **[gotcha]** When adding a sugarcraft library, transitive `sugarcraft/candy-core @dev` dependencies are not auto-discovered through nested VCS repositories — composer fails with "could not be found in any version". Explicitly add every transitive sugarcraft package to BOTH `require` and `repositories` (candy-core is required by nearly every other candy-* package).
- **[pattern]** Before running `gh` commands in this repo, `unset GH_TOKEN GITHUB_TOKEN` first. The shell environment has tokens set that don't authenticate against `detain/*` repos; unsetting falls back to the working keyring/SSH auth.
- **[convention]** The `/home/sites/mystage/include/Api/` directory (~530 PHP files, ~3,295 comparable blocks) is the canonical real-world benchmark corpus referenced in `docs/BENCHMARKS.md`. Use it (not `tests/Fixtures/`) when measuring perf changes — fixtures are too small to surface clustering hot-path effects.
- **[gotcha]** SugarCraft styling APIs (`Style::foreground()`, chip backgrounds, etc.) emit ANSI escapes even when wrapped in a `Theme::plain()` flow — `Theme::plain()` only neutralises the theme's own `success`/`error`/etc. styles, not arbitrary `Style::new()->foreground(...)` calls. For `--no-ansi` fidelity, gate explicit `->foreground()`/`->background()` calls behind a `$decorated` flag derived from `$output->isDecorated()`.
- **[correction]** When planning SugarCraft integration, do NOT scope research to `vendor/sugarcraft/` (which only has ~4 libs: candy-core, candy-kit, candy-palette, candy-sprinkles). The full monorepo at `/home/sites/sugarcraft/` ships 24+ libraries — sugar-bits (15 prebuilt components incl. Spinner/Progress/TextInput), sugar-charts (Sparkline/BarChart/LineChart/Heatmap), sugar-prompt (forms), candy-shine (Markdown→ANSI), candy-shell (gum CLI subcommands), sugar-veil (overlays), sugar-toast (notifications), sugar-stickers (FlexBox), sugar-boxer (panel composition), sugar-table, sugar-calendar, candy-vt, candy-zone, candy-log, candy-metrics, candy-wish, honey-bounce, etc. **Why:** picking the right primitive depends on knowing the full menu — e.g. don't custom-roll a spinner via `Cmd::every()` when sugar-bits already has `Spinner`. **How to apply:** when writing a SugarCraft-oriented plan or recommending a component, browse `/home/sites/sugarcraft/README.md` and the relevant `<lib>/README.md` files first; each subdir maps to `github.com/sugarcraft/<dirname>`.
- **[gotcha]** `composer.lock` is gitignored in this repo (`/composer.lock` on line 2 of `.gitignore`). After `composer update` / `composer require`, `git status` will only show `composer.json` — that's expected, not a missed file. Don't try to `git add composer.lock`; users intentionally rebuild from `composer.json` rather than ship a lock.
- **[convention]** SugarCraft PHP namespaces nest the component name twice: the autoloadable class for a library `sugarcraft/<lib>` lives at `SugarCraft\<Lib>\<Component>\<Component>` (e.g. `SugarCraft\Bits\Spinner\Spinner`, `SugarCraft\Charts\Sparkline\Sparkline`, `SugarCraft\Stickers\Flex\FlexBox`). Top-level shortcuts like `SugarCraft\Bits\Spinner` won't resolve — always use the fully-nested form when writing `use` statements or doc references.
- **[pattern]** When manually editing `composer.json` to add multiple sugarcraft packages, run `composer update <pkg1> <pkg2> ... --with-all-dependencies` rather than `composer require pkg1 pkg2 ...`. The `update` form respects all the manually-edited entries (incl. transitives added to fix "could not be found" errors) without re-running the resolver from scratch on each package.
- **[fix]** `composer require psalm/psalm` fails with "could not be found in any version" — the package name on Packagist is `vimeo/psalm`, not `psalm/psalm`. Use `composer require --dev vimeo/psalm` instead. (Composer's error message helpfully suggests the correct vendor.)
- **[gotcha]** PHP-Parser 5 (used in this repo) removed `PhpParser\Node\Stmt\Throw_` — `throw` is now an expression only. Use `Node\Stmt\Expression` whose `->expr` is `Node\Expr\Throw_`. Any `instanceof Node\Stmt\Throw_` check is dead code and triggers `phpstan class.notFound`.
- **[gotcha]** In PHP-Parser 5, `Node\VarLikeIdentifier` extends `Node\Identifier`. So `instanceof Node\Identifier` already matches both — a separate `instanceof Node\VarLikeIdentifier` branch placed after `Identifier` is unreachable (phpstan flags it `instanceof.alwaysFalse`). If you need the VarLikeIdentifier-specific behaviour, check the subclass FIRST, then `Identifier`.
- **[convention]** Static analysis runs at phpstan level 6 (`phpstan.neon`) and psalm `errorLevel=6` (`psalm.xml`), both scoped to `src/` only with `vendor/` and `tests/Fixtures/` excluded. Don't bump levels or widen scope without coordination — fixtures intentionally include malformed PHP to exercise parser error paths.
- **[pattern]** `vendor/bin/phpstan analyse --error-format=json` followed by piping into a small Python `json.loads` script gives a much more compact view of errors than the default formatter (which wraps lines at 80 cols and adds blog-link footers). Useful when triaging dozens of errors at once.
- **[fix]** Psalm's `ParadoxicalCondition` (psalm/089) and downstream `InvalidCast` (`never cannot be cast to int`) on repeated `array_key_exists($k, $data)` checks across separate `if` blocks are caused by Psalm narrowing the array shape after the first positive check (it then considers later positive checks contradictory). Fix: cache existence once into local booleans (`$hasMin = array_key_exists('min_block_size', $data); $hasMax = array_key_exists('max_block_size', $data);`) and reuse those in subsequent guards. Psalm treats the booleans opaquely and stops fighting the narrowing.
- **[pattern]** Smoke-test `--validate-config` and similar config-gating CLI flags via a tiny inline shell script that writes a JSON to `mktemp --suffix=.json`, runs the command, prints `exit: $?`, then mutates the file for invalid/missing-flag cases. Inspecting exit codes per branch (0 valid / 2 invalid / 2 misuse) catches regressions in argv plumbing that PHPUnit can't see.
- **[pattern]** When wiring a new layered config source into `Phpdup\Cli\ConfigLoader::load()` (e.g. profile excludes), pass it as a separate named parameter and apply it ONLY when the user-facing source (`--config` JSON or matching `$overrides[$key]`) is empty. Excludes specifically resolve as `!empty($data['exclude']) ? $data['exclude'] : ($profileExclude ?? $base->exclude)` — keep profiles as a *default upgrade*, not a forced override, so `--config` and explicit CLI flags continue to win.
- **[convention]** Profile JSON files in `/home/sites/php-dup-finder/profiles/*.json` use a `_description` key for human-readable docs that `ProfileRegistry::load()` strips before returning, because the validator rejects unknown keys. When adding a new profile, keep `_description` for humans but never reference it from code — it won't survive the loader.
- **[convention]** Profile JSON files use `kinds` as the user-facing key but `Config` / `$overrides` use `allowed_kinds` internally. The mapping happens in `Command::profileToOverrides()` — when adding a new profile-mappable key, mirror that translation rather than passing the JSON dict straight through to `ConfigLoader`.
- **[gotcha]** When phpunit reports `OK (266 tests)` but `phpstan analyse` then prints `[ERROR] Found 2 errors`, the chained `&& ... | tail -5` swallows the actual error lines — phpstan's table formatter pushes the bodies above the `tail -5` window. Re-run `vendor/bin/phpstan analyse 2>&1 | tail -50` (or use `--error-format=json`) to see what failed; never declare "all clean" off a tail-5 of a chained run.
- **[pattern]** When adding a new disk-cache layer to phpdup (e.g. `IndexStore`, `ClusterCache`), gate enablement behind `$config->incremental` AND exclude modes where the cache layer adds nothing — `ClusterStage` excludes `exactOnly` because hash-bucket clustering is already O(n). Build the cache config-key from the fields that actually drive output (similarity/tree thresholds, optional-blocks tunables, `exactOnly`); mismatched key invalidates wholesale. Strip heavy `PhpParser` AST nodes from cluster payloads before `serialize()` (set `$copy->generalizedAst = null` on a clone) — they bloat the cache by ~10x and are reconstructed on demand by `RefactorStage`.
- **[pattern]** Smoke-test new disk-cache layers with this 4-step shell: `rm -rf .phpdup-cache; php bin/phpdup analyze <fixture> --json /tmp/c1.json; php bin/phpdup analyze <fixture> --json /tmp/c2.json; diff /tmp/c1.json /tmp/c2.json && ls .phpdup-cache/`. The first run populates the cache; the second must hit it (look for `reused N clusters from cache` or equivalent log line); the diff must be empty (deterministic output across cache hit/miss); the `ls` confirms the new cache file lives alongside existing ones (`*.idx`, `*.cache`, `clusters.list`).
- **[gotcha]** `serialize()` of `Cluster` objects round-trips fine with `unserialize($blob, ['allowed_classes' => true])` only because `Cluster` is `final` with public mutable properties. If you mark fields `readonly`, `unserialize` cannot rehydrate them through the normal property-write path and you'll need to switch to `__serialize`/`__unserialize` explicitly.
