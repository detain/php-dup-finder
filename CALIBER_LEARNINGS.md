# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** When piping cluster results through a post-Ranker enrichment stage that mutates `Cluster` properties (e.g. `CoherenceAnalyzer::analyze()` populating `$outlierMemberIds`), run it inside `ReportStage::execute()` AFTER `Ranker::rank()` — Ranker filters and reorders, so doing the enrichment before wastes work on clusters about to be dropped, and breaks downstream reporters that rely on indices in the final ordering.
- **[convention]** `Block` properties (`class`, `file`, `namespace`, `name`, `ast`, `ngramBag`, `id`) are publicly mutable — tests deliberately overwrite them post-construction to exercise pattern-recognizer / coherence / index branches keyed off file paths, class-name suffixes, n-gram shape, or block ids. Keep them mutable; if you ever propose marking them `readonly`, every reporter/pattern/index test in `tests/Unit/Refactor/`, `tests/Unit/Reporting/`, and `tests/Unit/Index/` will need to be rewritten.
- **[pattern]** Tests for analyzers that consume `Block::$ngramBag` (e.g. `CoherenceAnalyzer`, anything calling `JaccardSimilarity::similarity()`, `BloomCandidateIndex`) can skip the full extractor pipeline: `$stmts = (new AstParser())->parseCode('<?php $x;'); $b = new Block(file: 'x.php', range: new LineRange(1, 1), kind: 'method', namespace: null, class: null, name: null, ast: $stmts[0]); $b->ngramBag = $bag; $b->id = 'A';` is enough because `Block`'s ngram-related properties are publicly mutable. Faster than re-running `BlockExtractor` + `NgramFingerprint` per fixture.
- **[gotcha]** The classic SWAR popcount (`$x - (($x >> 1) & 0x5555…)` ... `($x + ($x >> 4)) & 0x0F0F…`) does NOT translate to PHP — PHP ints are signed-only, so for sign-bit-heavy 64-bit inputs the subtraction underflows and the result silently widens to float, triggering `Implicit conversion from float … to int loses precision` deprecations under `phpunit.xml`'s `failOnNotice=true`. Use a per-byte 256-entry lookup table (build once into a `static $table`, then sum eight `$table[($x >> N) & 0xFF]` lookups) — same correctness, no underflow, fast enough for filter-overlap inner loops. PHPStan/phpunit catch this only when actual sign-bit-set words are exercised, so add a test that constructs a filter likely to set the high bits.
- **[gotcha]** PHPUnit 10's terse default report says `OK, but there were issues!` with just `Deprecations: N` — to see WHICH deprecations and WHERE, re-run with `--display-deprecations` and `head -40` (the per-issue blocks come at the TOP of the output, before the summary). `tail` and `sed` filters miss the message bodies because phpunit puts the file/line on one line and the message text on the next. Also: `phpunit.xml` sets `failOnNotice=true` and `failOnWarning=true`, so any `Implicit conversion from float … loses precision` notice WILL turn green tests red downstream — fix at the source rather than suppressing.
- **[pattern]** When testing a Bloom-filter-like overlap index, tiny test bags (3-4 ngrams) produce noisy false positives because the FPR floor dominates the signal. Build realistic-sized fixtures with `array_fill_keys(array_map(fn($i) => "shared-$i", range(1, 30)), 1)` so the overlap-vs-disjoint contrast is unambiguous — a `notContains` assertion against a 3-element disjoint bag will flap.
- **[pattern]** When adding a new disk-cache layer to phpdup (e.g. `IndexStore`, `ClusterCache`, `TokenCache`), gate enablement behind `$config->incremental` AND exclude modes where the cache layer adds nothing — `ClusterStage` excludes `exactOnly` because hash-bucket clustering is already O(n). Build the cache config-key from the fields that actually drive output (similarity/tree thresholds, optional-blocks tunables, `exactOnly`); mismatched key invalidates wholesale. Strip heavy `PhpParser` AST nodes from cluster payloads before `serialize()` (set `$copy->generalizedAst = null` on a clone) — they bloat the cache by ~10x and are reconstructed on demand by `RefactorStage`.
- **[pattern]** Smoke-test new disk-cache layers with this 4-step shell: `rm -rf .phpdup-cache; php bin/phpdup analyze <fixture> --json /tmp/c1.json; php bin/phpdup analyze <fixture> --json /tmp/c2.json; diff /tmp/c1.json /tmp/c2.json && ls .phpdup-cache/`. The first run populates the cache; the second must hit it (look for `reused N clusters from cache` or equivalent log line); the diff must be empty (deterministic output across cache hit/miss); the `ls` confirms the new cache file lives alongside existing ones (`*.idx`, `*.cache`, `*.tok`, `clusters.list`).
- **[gotcha]** `serialize()` of `Cluster` objects round-trips fine with `unserialize($blob, ['allowed_classes' => true])` only because `Cluster` is `final` with public mutable properties. Same applies to `PhpParser\Token` (since php-parser 5.0 they're plain value objects so `TokenCache` can serialize them). If you mark fields `readonly`, `unserialize` cannot rehydrate them through the normal property-write path and you'll need to switch to `__serialize`/`__unserialize` explicitly.
- **[pattern]** Tests for analyzers that consume `Block::$ngramBag` (e.g. `CoherenceAnalyzer`, anything calling `JaccardSimilarity::similarity()`, `BloomCandidateIndex`) can skip the full extractor pipeline: `$stmts = (new AstParser())->parseCode('<?php $x;'); $b = new Block(file: 'x.php', range: new LineRange(1, 1), kind: 'method', namespace: null, class: null, name: null, ast: $stmts[0]); $b->ngramBag = $bag; $b->id = 'A';` is enough because `Block`'s ngram-related properties are publicly mutable. Faster than re-running `BlockExtractor` + `NgramFingerprint` per fixture. Same trick works for `TokenCache`/`AstCache` round-trip tests — synthesize `new \PhpParser\Token(\T_OPEN_TAG, '<?php ', 1, 0)` instead of running the real lexer.
- **[convention]** `Block` properties (`class`, `file`, `namespace`, `name`, `ast`, `ngramBag`, `id`, `rangeHash`) are publicly mutable — tests deliberately overwrite them post-construction to exercise pattern-recognizer / coherence / index / incremental branches keyed off file paths, class-name suffixes, n-gram shape, block ids, or per-block source hashes. Keep them mutable; if you ever propose marking them `readonly`, every reporter/pattern/index test in `tests/Unit/Refactor/`, `tests/Unit/Reporting/`, and `tests/Unit/Index/` will need to be rewritten.
- **[env]** Always prefix `gh` commands with `unset GH_TOKEN GITHUB_TOKEN` (or run them in a subshell that does) — the local shell exports tokens that conflict with the credentials `gh auth login` stores, producing silent auth failures or commands that hit the wrong account. The pattern `unset GH_TOKEN GITHUB_TOKEN; gh pr create ...` is used for every gh invocation in this repo.
- **[gotcha]** `gh pr list` (no flags) only shows OPEN PRs and prints nothing when none are open — the empty output is success, not a failure. Use `gh pr list --state all --limit 10` to see recently merged PRs (e.g. when verifying a `gh pr merge --delete-branch` actually closed the right number).
- **[pattern]** When adding a new top-level field to `JsonReporter::build()` output (e.g. `schema_version`, `architectural_findings`), bump `Phpdup\Reporting\JsonSchemaSpec::SCHEMA_VERSION` per its semver-ish rules: a new optional field is MINOR (`1.0` → `1.1`); a removed/renamed field or type change is MAJOR (`1.x` → `2.x`). External consumers (the JetBrains plugin in particular — see `docs/JETBRAINS_PLUGIN.md`) refuse to load on MAJOR mismatches.
- **[pattern]** When wiring a new analyzer-style enrichment (e.g. `CoherenceAnalyzer`, `SolidAnalyzer`, `DesignPatternAnalyzer`, `AntiPatternAnalyzer`) into `ReportStage::execute()`, instantiate the analyzers as a single `[$a, $b, $c]` array AFTER `Ranker::rank()` and iterate them per-cluster into a single `Cluster::$architecturalFindings`-style list field. Keep the field publicly mutable on `Cluster` (don't `readonly`-it) so multiple analyzers can append over the loop. Then surface in `JsonReporter::build()` and `CliReporter::renderCluster()` so the new findings show up everywhere — and remember the Golden snapshots will need `UPDATE_SNAPSHOTS=1 vendor/bin/phpunit --testsuite Golden`.
- **[gotcha]** `Phar::running()` argument semantics are the OPPOSITE of intuition: `Phar::running(true)` returns the `phar://` stream URL (`phar:///abs/path/foo.phar`), `Phar::running(false)` returns the on-disk path (`/abs/path/foo.phar`). To build a clean `phar://<path>/vendor/autoload.php` URL inside a phar stub, call `Phar::running(false)` and prepend `phar://` yourself — calling `Phar::running(true)` produces a doubled `phar://phar:///...` URL that fails to resolve with the misleading error `phpdup: autoload not found inside phar`. Same trap applies to `build-phar.php`'s generated stub.
- **[pattern]** When hardening `unserialize($blob, ['allowed_classes' => true])` paths (e.g. `AstCache`, `ClusterCache`, `IndexStore` in `src/Persistence/`), do TWO things: (1) replace `true` with an explicit `SerializedClassAllowList::*Classes()` whitelist so unexpected classes decode as `__PHP_Incomplete_Class`, and (2) post-validate by walking the deserialized payload (including one level of nested arrays where the cache stores `Block[]` etc.) and treating any `__PHP_Incomplete_Class` instance as a cache miss — return `null` rather than letting an incomplete object reach the pipeline. The allow-list lives in `src/Persistence/SerializedClassAllowList.php` and enumerates parser classes via composer's classmap with a directory-walk fallback when the project was installed without `-o`.
- **[gotcha]** Running `vendor/bin/phpunit` (or any vendor binary) from a fresh checkout fails with `No such file or directory` because `composer.lock` is gitignored and `vendor/` is not committed. Always `composer install --no-interaction` first; the noisy `Class Fixtures\Notify\Notify ... does not comply with psr-4 autoloading standard` warning is expected (intentional malformed fixtures under `tests/Fixtures/notify/`) and not a failure.
- **[gotcha]** `gh pr list` (no flags) only shows OPEN PRs and prints nothing when none are open — the empty output is success, not a failure. Use `gh pr list --state all --limit 10` to see recently merged PRs (e.g. when verifying a `gh pr merge --delete-branch` actually closed the right number). Also: `gh pr view --json …` (no `<number>`) errors with `no pull requests found for branch "<current>"` when the current branch has no PR — pass an explicit PR number or use `gh pr list --head <branch>` instead.
- **[pattern]** When recreating a feature branch that already exists locally (e.g. left over from a prior `git checkout -b` attempt), `git checkout -b <branch>` fails with `fatal: a branch named '<branch>' already exists`. Recovery: `git branch -D <branch>; git checkout -b <branch>` then RE-STAGE the index (the second checkout dropped what you'd `git add`-ed). Don't try to be clever with `git checkout -B <branch>` — it preserves the index but silently moves the existing branch ref, which can clobber unpushed commits on the prior attempt.
- **[pattern]** When adding `bench/` artifacts: track scripts (`bench/*.php`, `bench/run-all.sh`), the hand-curated `feature-matrix.md`, and `bench/corpora/synthetic-fuzz/.ground-truth.json`; gitignore the regeneratable bits (`/bench/tools/`, the per-corpus directories under `/bench/corpora/<oss-name>/`, `/bench/corpora/synthetic-fuzz/*.php`, `/bench/results/*.json`, `/bench/results/*.md`). Validate the patterns before committing with `git check-ignore -v bench/corpora/<a-keep-file> bench/corpora/<an-ignore-file>` to confirm both sides of the split match.
- **[gotcha]** `src/Parallel/PreprocessWorker.php` caches per-config tooling (extractor + normalizer + ngram fingerprinter) keyed by a `sprintf()` fingerprint of the Config fields that affect normalization output. When adding a new `Config` field that the Normalizer reads (e.g. `normalizationPlugins`, `dbAware`), append it to the cache-key `sprintf()` — otherwise stale tooling is reused across configs differing only by the new field, producing wrong-canonical-form blocks under per-directory overrides. Same applies to `src/Pipeline/Stages/PreprocessStage.php`'s config-key hash used by `IndexStore`: any new normalization-affecting Config field must be folded in or the disk cache returns stale-canonicalised blocks when the flag is toggled between runs.
- **[pattern]** Adding a new top-level config JSON key (e.g. `normalization`, `db_aware`) requires THREE updates to `src/Cli/ConfigLoader::validate()`: append the key to the allowed-keys whitelist, add a per-key validation block (object/boolean type-check + nested-keys whitelist for objects), and call the existing `$assertListOfStrings()` helper on any list-of-string sub-fields. Mirror an existing block of the matching shape — `report` for object keys, an existing boolean field for scalars. Then add at least four tests in `tests/Unit/Cli/ConfigLoaderTest.php`: defaults-off (or default value), reads from config file, override-beats-config-file, and validation-rejects-wrong-type.
- **[pattern]** When adding a new disk-cache layer to phpdup (e.g. `IndexStore`, `ClusterCache`, `TokenCache`), gate enablement behind `$config->incremental` AND exclude modes where the cache layer adds nothing — `ClusterStage` excludes `exactOnly` because hash-bucket clustering is already O(n). Build the cache config-key from the fields that actually drive output (similarity/tree thresholds, optional-blocks tunables, `exactOnly`, any new normalization-affecting flag like `dbAware`); mismatched key invalidates wholesale. Strip heavy `PhpParser` AST nodes from cluster payloads before `serialize()` (set `$copy->generalizedAst = null` on a clone) — they bloat the cache by ~10x and are reconstructed on demand by `RefactorStage`.
- **[pattern]** When adding a `Normalizer` pre-pass that runs *before* the standard variable/literal/name canonicalisation (e.g. `DbOpCanonicalizer` for `--db-aware`), structure it as a separate visitor class in `src/Normalization/` with a registry sidecar (e.g. `DbOpRegistry`) and gate it on a Config boolean so `Normalizer::__construct()` can no-op the pass when off. The pre-pass's synthetic FuncCall names (`__DB_<OP>__`) survive the aggressive name-canonicalisation pass because they're treated as structural function names like `isset`/`count`. Cache invalidation: thread the new flag through BOTH `src/Parallel/PreprocessWorker.php`'s tooling cache-key sprintf AND `src/Pipeline/Stages/PreprocessStage.php`'s IndexStore config-key hash.
- **[pattern]** Adding a new top-level config JSON key (e.g. `normalization`, `db_aware`, `ml_pair_url`) requires THREE updates to `src/Cli/ConfigLoader::validate()`: append the key to the allowed-keys whitelist, add a per-key validation block (object/boolean/string/number type-check + nested-keys whitelist for objects), and call the existing `$assertListOfStrings()` helper on any list-of-string sub-fields. Mirror an existing block of the matching shape — `report` for object keys, an existing boolean/number for scalars, `ml_url` for SSRF-validated URL fields. - **[gotcha]** In this environment `grep` resolves to `ugrep`, which parses any pattern that begins with `-` as an option — `grep -rn "->run("` fails with `ugrep: invalid option ->run(` (seen twice this session). Pass the pattern with `-e` (`grep -rne '->run('`), put `--` before it, or anchor on a non-dash substring (`grep -rn 'pool->run('`).
- **[gotcha]** `AstSerializer::shortType()` emits **prefixed** PhpParser labels (`Expr_MethodCall`, `Stmt_If_`, `Scalar_Int_`) — NOT the bare names (`MethodCall`, `If_`, `Int_`). Any similarity/cost code that matches on bare labels (e.g. `EditCostModel::cost()`) silently no-ops against real input, and unit tests that hand-write bare labels pass green while never exercising the real path. When testing label-keyed logic, feed the actual `shortType()` output (confirm it with a `php -r` reflection one-liner that parses a snippet and prints `shortType()`), not hand-written strings, and assert *strict* behavioral differences (not `assertLessThanOrEqual`).
- **[pattern]** To empirically confirm a `src/` class's runtime behavior from the shell, run `php -r 'require "vendor/autoload.php"; ...'`. Watch for static-factory assumptions: `EditCostModel` has no `::semantic()` factory (calling it fatals and Bash returns an empty `{}` result) — construct it as `new EditCostModel('semantic')`. An empty `{}` Bash response generally means the PHP fataled, not that the search found nothing — re-check the API before concluding.
- **[gotcha]** When adding `Match_`/`Switch_` normalization to `Normalizer`, ensure the transformation feeds forward into clustering: both `Expr\Match_` and `Stmt\Switch_` must normalize to the same `__MATCH__(subject, cond, body, ...)` FuncCall shape so the hasher/fingerprinter sees identical fingerprints. Normalizing to structurally different ASTs (e.g. an `If_` chain) will NOT produce matching hashes — the whole point is that `__MATCH__` is the common canonical form. Also watch for dead code in old normalization helpers: a `$cases` array built but never returned or stored is a sign the helper was scaffolding work that was never wired up.
- **[env]** When scripting `gh` in this workspace, `unset GITHUB_TOKEN GH_TOKEN` first (the user explicitly added this to the remediation plan): a stray exported token overrides the `gh` keyring login and is commonly wrong-scoped/expired, breaking `gh pr create`/`merge` with auth errors. Each freshly-spawned agent shell needs the `unset` re-run; verify with `gh auth status` afterward.
